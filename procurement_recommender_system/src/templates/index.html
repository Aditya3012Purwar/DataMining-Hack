How the Recommendation Engine Works
Two-stage recommender system

With 2.7M SKUs and 18M transactions, this uses a candidate generation + ranking pipeline — the same architecture as Amazon/Netflix.

Stage 1 — Candidate Generation

The buyer's E-class purchase history filters the catalogue to same-category, not-yet-purchased items. Shrinks millions to a manageable pool.

Stage 2 — Ranking

final_score = P(buy) x (1 + price_advantage)
P(buy) combines: global popularity + manufacturer familiarity + price advantage + feature similarity + feature richness + collaborative filtering

Feature-based matching (key innovation)

We build a customer feature profile by aggregating all feature key-value pairs (color, size, material, dimensions, etc.) from the buyer's previously purchased products (using fvalue_set — the curated, consolidated values). Each candidate is then scored by Jaccard similarity: how much its features overlap with what the customer has bought before. This ensures recommendations match the customer's actual preferences, not just category.

fvalue vs fvalue_set

fvalue is the raw supplier data (may be messy/incomplete). fvalue_set is the curated version where values are consolidated across all SKUs in a set_id. We prefer fvalue_set.

What "Saved %" means

Saved % = (ref_price - candidate_price) / ref_price
The reference price is this buyer's own average paid price for the same E-class — not a global average.

Column meanings
  • Cand. EUR — average market price of the alternative item
  • Ref. EUR — what this buyer currently pays on average for this E-class
  • Saved % — actual price difference vs buyer's own reference
  • Feat. Sim. — feature similarity: Jaccard overlap with customer's purchased features (0-100%)
  • Collab — collaborative filtering: co-purchase affinity between E-classes (0-100%)
  • Cluster — feature cluster ID grouping similar products by shared features
  • P(buy) — estimated purchase probability (0-1)
  • Score — combined rank: higher = more likely AND cheaper
